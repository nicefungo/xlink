## 必须执行的每日调度

**按以下顺序执行，做完一步再下一步。必须执行到最后一步。**

---

### Step 1: 构建验证
```bash
cd /home/admin/xlink && make clean && make all && make test
```
- 如果 `make all` 失败 → 修复后重新构建
- 如果 `make test` 有 FAIL → 修 bug（进入 Step 5）
- 记录结果

---

### Step 2: 推进未来规划文档

检查 `docs/future-plans/` 目录，**必须有实质性推进**。

优先顺序：
1. **如果有未完成的 phases**（检查 `02-async-io-phases.md` 等）→ 实现下一个步骤（写代码 + 测试）
2. **如果所有 phases 已完成** → 选择一个 future-plans 文档，做以下之一：
   - 深化设计方案（补充接口细节、性能分析、伪代码）
   - 调研相关技术（搜索 C 项目最佳实践、阅读 Linux 内核文档、参考同类库设计），将发现写入 plan 文档
   - 新增一个 future-plan 方向（新的架构改进想法）
   - 更新 index.md 路线图、依赖关系、决策日志

**每个计划文档的结构：**
```markdown
# 标题

## 动机
## 设计方案
## 实现路径 (Phase 1/2/3)
## 依赖
## 开放问题
## 关联文档
```

**你不能报告"无待推进步骤"然后什么都不做。第二步必须产出内容。**

如果所有方向都穷尽了 → 从以下方向中找一个做：
- 性能基准测试分析（对比 epoll vs io_uring vs poll 的延迟/吞吐量）
- 内存分配器优化方案（slab 替代品？arena allocator？）
- 跨平台兼容性调研（Windows I/O completion ports、嵌入式 RTOS）
- 错误处理策略改进
- API 易用性改进（helper macros、simplified init）
- 搜索同类 C 通信库（zeromq、nanomsg、nng），对比分析，记录到 future-plans

---

### Step 3: 更新现有文档

检查涉及今天改动的文档是否过时。如果今天改了代码 → 更新对应文档。

---

### Step 4: [每周] 代码审查 + 补充测试

**仅在以下情况执行**（否则跳过）：
- 今天是周日
- 或者 src/ 有变更

审查内容：
- 通看 `/home/admin/xlink/src/` 新增/修改的代码（资源泄漏、竞态、返回值检查）
- 补至少一个测试，覆盖错误路径或边界值

---

### Step 5: 修 bug
确认的 bug 直接修。修完验证 `make test` 全过。

---

### Step 6: 进度自检 🚨

**两个判断，任何一条不通过 → 回到 Step 1 继续工作。**

**判断 1: 有没有实质性进度？**
- 今天有没有：代码改动、文档更新、或新研究/想法记录？
- 没有 → 回到 Step 1，输出消息：
  > **"Do something! Make progress in code, or update some points in future plans, or search and research for ideas of what to do!"**

**判断 2: 进度够不够形成一个 commit？**
- 检查 `git status`：改动是否足够有意义（不是仅改了几个空格或一句注释）？
- 不够 → 回到 Step 1

**判断 3: 距离上次代码变更是否超过 2 天？**
- 检查 `git log --oneline -30`，找到最后一次 src/ 代码变更（`src/*.c`、`include/*.h`、`tests/*.c` 的修改，非纯文档 commit）
- 如果距离上次代码变更已经 ≥ 2 天 → **必须产出代码变更！** 文档更新不算。从 Step 2 开始重跑，直到有代码产出为止
- 代码产出定义：新增或修改 `src/*.c`、`include/*.h`、或 `tests/*.c`（测试也算代码）
- 如果最近 2 天内有代码变更 → 文档/研究也算有效进度

**重试规则：重试一次后仍无进展 → 记录原因，不再重试，直接进入 Step 7。**

如果触发重试，输出消息：
> **"Do something! Make progress in code, or update some points in future plans, or search and research for ideas of what to do!"**

---

### Step 7: 文档一致性检查 📋

核心原则：文档必须反映代码的当前状态。不一致 → 更新文档（不要改代码匹配文档）。

**每日（周一~周六）：** 只检查今天改动涉及的文档

**周日：** 全量检查 `/home/admin/xlink/docs/` 下所有 .md 文件，出具报告

| 文档 | 对标来源 | 检查项 |
|------|----------|--------|
| `integration-guide.md` | `xlink.h` | API 签名、常量、结构体 |
| `api.md` | `xlink.h` | 所有公开 API |
| `proposal.md` | `src/` | 设计描述 vs 实际实现 |
| `slab-allocator.md` | `src/xlink.c` | Slab 结构、宏、函数 |
| `known-issues.md` | 最新测试结果 | 已修 bug 移入已修复列表 |
| `code-walkthrough.md` | `src/` + `include/` | 架构/模块描述 |
| `design-decisions.md` | 代码实现 | 决策是否仍成立 |
| `future-plans/*.md` | 代码状态 | 已完成项打标，新增想法 |

---

### Step 8: 提交与推送 📤

```bash
cd /home/admin/xlink
git add -A
# 如果没有改动，跳过提交
git diff --cached --quiet && echo "Nothing to commit" || \
  git commit -m "daily: $(date +%Y-%m-%d) - <简短描述>"
git push origin master
```

- commit message 必须包含日期 + 一句话描述
- push 必须成功；如果失败，排查原因并重试

## 核心约束（必须遵守）

- ❌ 不做大改动/重构（只写计划，除非是实现 phases）
- ❌ **绝对禁止报告"无待推进步骤"后什么都不做**
- ✅ 修 bug 随时做
- ✅ **每天必须有实质性进度（Step 6 自检）**
- ✅ **每天必须 push 到 GitHub（Step 8）**
- ✅ **周二优先做代码审查 + 测试补充**
- 🔄 如果所有 phases 完成且所有计划穷尽 → 做调研、写对比分析、找新方向
